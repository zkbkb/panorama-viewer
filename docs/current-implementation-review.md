# 当前实现审查与优化建议（Panorama Viewer）

> 审查范围：`src/` 目录下的渲染、输入、陀螺仪、上传与控制层实现。

## 总体结论

当前实现已经具备良好的基础能力：

- Three.js 全景渲染流程清晰，生命周期管理基本完整。
- 手势与陀螺仪融合思路正确，支持 recenter/horizon lock 等核心能力。
- UI 结构简洁，移动端优先策略明显。

但在**可靠性、性能自适应、可访问性、错误反馈、可测试性**方面仍有明显提升空间。以下建议按优先级排序。

---

## P0（优先立刻优化）

### 1) 图片加载失败没有用户可感知反馈
- 现状：纹理加载失败时仅 `setIsLoading(false)`，没有错误提示与恢复路径。参考 `usePanoramaRenderer` 的 loader error callback。
- 风险：用户会看到“无响应/黑屏”，不知道图片格式问题、损坏或跨域问题。
- 建议：
  - 增加 `loadError` 状态与 Overlay 错误提示（例如“加载失败，请尝试其他文件”）。
  - 提供“返回重新上传”按钮，降低失败路径摩擦。

### 2) 上传校验过于宽松，缺少大小/像素维度保护
- 现状：`UploadScreen` 仅按 `file.type.startsWith("image/")` 过滤。
- 风险：超大图（例如 100MB、超高分辨率）可能导致内存峰值过高、移动端崩溃。
- 建议：
  - 上传前做文件大小上限（如 20MB）与最小推荐分辨率提示。
  - 加上 `createImageBitmap` 预读取尺寸，超限提示后拒绝进入渲染阶段。

### 3) 无障碍语义不完整（按钮/状态缺少 ARIA）
- 现状：控制按钮主要依赖 `title`，没有 `aria-label` / `aria-pressed`。
- 风险：屏幕阅读器体验差，状态型按钮（gyro、horizon lock）不可理解。
- 建议：
  - 为所有 icon button 增加 `aria-label`。
  - toggle 类按钮增加 `aria-pressed={true/false}`。

---

## P1（高价值改进）

### 4) 渲染质量应按设备性能自适应
- 现状：球体几何固定 `SphereGeometry(500, 64, 32)`，像素比上限固定 2。
- 风险：低端设备掉帧，高端设备细节提升空间不足。
- 建议：
  - 根据 `navigator.hardwareConcurrency`、屏幕尺寸或首帧耗时动态降级/升级：
    - 低端：`32x16` + DPR 1.25
    - 中端：`48x24`
    - 高端：`64x32` + DPR 2

### 5) FOV 与交互速度参数应配置化
- 现状：`MIN_FOV/MAX_FOV/DEFAULT_FOV/LERP_FACTOR` 常量硬编码。
- 风险：后期调参与 A/B 实验成本高。
- 建议：
  - 提取到统一 `viewerConfig`，支持按设备类型覆盖。
  - 预留 URL query 覆盖（如 `?fov=80`）用于 QA 快速验证。

### 6) 陀螺仪可用性检测仍偏乐观
- 现状：`isAvailable` 仅判定 `window.DeviceOrientationEvent` 是否存在。
- 风险：部分浏览器“存在 API 但无权限/无传感器”，按钮可见但不可用。
- 建议：
  - 增加“是否收到过有效 orientation 数据”的二次判定。
  - 若多次启用失败，提示“设备/浏览器不支持”。

### 7) 全屏能力缺少兼容回退策略
- 现状：只使用标准 Fullscreen API，异常被静默吞掉。
- 风险：用户点击无反馈。
- 建议：
  - 返回操作结果（成功/失败）并在失败时 toast 说明原因。
  - 可选兼容：iOS Safari 下给出“添加到主屏幕”替代建议。

---

## P2（体验打磨）

### 8) 上传页缺少“示例图/格式说明”与错误提示
- 现状：仅文字提示支持 JPG/PNG/WebP。
- 建议：
  - 增加“什么是等距柱状图（equirectangular）”的简短说明。
  - 不符合格式时弹出明确提示，而非静默忽略。

### 9) 手势体验可进一步细腻化
- 现状：拖拽/缩放比例固定。
- 建议：
  - 基于当前 FOV 调整拖拽灵敏度（缩小时降低灵敏度更自然）。
  - 增加 inertial damping（惯性）可提升“丝滑感”。

### 10) hint 与 gyro 引导策略可个性化
- 现状：提示时长固定；每次进入 viewer 都会提示一次。
- 建议：
  - 使用 `localStorage` 记录“是否看过引导”。
  - 首次显示完整引导，后续改为简短版本。

---

## P3（工程与可维护性）

### 11) `usePanoramaRenderer` 职责较重，建议拆分
- 现状：单 hook 同时负责：场景初始化、纹理加载、输入事件、动画循环、陀螺仪融合。
- 风险：理解成本高，回归风险增大。
- 建议：拆为：
  - `useThreeScene`（渲染初始化与销毁）
  - `usePointerControls`（手势与 FOV）
  - `useGyroCamera`（姿态融合与水平锁定）

### 12) 缺少关键单元测试/集成测试
- 建议最小覆盖：
  - `orientation.ts` 的姿态转换正确性（核心数学逻辑）。
  - 上传校验逻辑（类型、大小、异常路径）。
  - 控件显隐逻辑（gyro 可用/启用状态）。

### 13) 缺少性能观测点
- 建议：
  - 记录纹理加载耗时、首帧时间、平均 FPS（可选 dev-only）。
  - 为线上问题定位保留基础日志开关。

## 我认为“性价比最高”的前三项
1. **失败可视化反馈**（直接降低用户困惑和流失）
2. **上传前限制与提示**（避免低端设备崩溃）
3. **按钮 ARIA 与状态语义**（可访问性收益大、改造成本低）
